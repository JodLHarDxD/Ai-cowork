JODL Co-Work System

A multi-model AI orchestration system for collaborative software development. Multiple AI providers (Claude, Gemini, GPT) each handle their domain specialty, passing outputs through a shared filesystem bus — keeping each model's context window small, focused, and token-efficient.

What Is This System

JODL is a multi-model AI co-work system built on a filesystem command bus. The core innovation: instead of sending one massive prompt to one AI, break the problem into domain-specialized tasks and route each task to the model best suited for it — with only the relevant context, not everything.

The token-efficiency insight: A single monolithic prompt for "build a luxury e-commerce page" might need 50K tokens of context. Split across 4 specialized agents (uiux → motion → typography → frontend), each agent gets ~5K tokens of domain-relevant context. Same outcome, 4× cheaper, each model is sharper in its lane.

04

Command Bus

The bus is pure filesystem — no network, no daemon, no database. Works across all AI platforms because they all have file access to D:\.agents\command-bus\.

Lock Model

Atomic rename = lock. pending-XXXX.yaml → claimed-<provider>-XXXX.yaml is atomic on both Windows (NTFS) and POSIX. If two providers race for the same task, one rename succeeds, the other fails silently — clean distributed lock with no coordination overhead.

Task Lifecycle

STATE 1
pending
Written by orchestrator. Has role, brief, provider, deps.
pending-XXXX.yaml
STATE 2
claimed
Atomic rename. Only one provider wins the race.
claimed-claude-XXXX.yaml
STATE 3
executing
Agent runs (API call or manual). Output streams to terminal.
in progress
STATE 4
done
Output written to XXXX.out.md. Rename to done-XXXX.yaml.
done-XXXX.yaml + .out.md

Session Context Accumulation

When any agent claims a task, the bus loads ALL done-*.out.md files from the session and injects them as "prior session outputs" into the agent's context. This means each agent sees everything that came before it.

⚠️ Current limitation: This is ALL outputs, not filtered to domain relevance. uiux-master doesn't need the database schema output, but gets it anyway. See Improvements for the fix.

Orchestrator Spawn Chain

When an orchestrator completes, the bus parses a JSON block at the end of its output:

// Orchestrator output must end with:
{
  "phase": "build",
  "spawn-tasks": [
    {
      "role": "frontend-master",
      "brief": "implement the CartDrawer component from the design spec",
      "depends-on": [],
      "parallel-group": 1
    }
  ],
  "ceo-report": "Design phase complete. 4 implementation tasks spawned."
}

Bus auto-creates child task YAML files. Next jodl next call by the right provider picks them up.

CLI Commands

# Set your identity
$env:JODL_PROVIDER = "claude-code"  # or: antigravity | codex

jodl whoami                   # verify identity + active sessions
jodl brief "build X"          # create session → spawn master-orchestrator task
jodl next                     # claim + run next task for your provider (auto-API for Claude)
jodl next --dry-run            # see what's next without claiming
jodl status                   # show all sessions + queue state
jodl claim <taskId>           # manual: get brief + system prompt (for Gemini/GPT providers)
jodl submit <taskId> -f out.md # manual: submit AI output from file
jodl watch                    # live terminal dashboard (run in separate terminal)

Gaps & Missing Pieces

Ordered by impact on production readiness. P1 = blocks core value. P2 = major UX pain. P3 = quality improvement. P4 = nice-to-have.

Multi-provider auto-execution
jodl next only auto-runs Claude tasks via Anthropic SDK. Antigravity (Gemini) and Codex (GPT) require a human to manually copy the brief, run it in their respective AI, and paste output back via jodl submit. This breaks the "autonomous co-work" promise. 2 of 3 providers are still manual.

Fix: Add @google/generative-ai SDK for Gemini tasks. Add openai SDK for GPT tasks. In bus-commands.ts next(), branch on task.assignedProvider to call the right client.
bus-commands.ts:~197
Context scoping (the token-efficiency core feature)
loadSessionContext() in bus.ts injects ALL prior task outputs into every agent. A frontend-master gets the uiux spec + motion spec + typography spec + schema + threat model + everything. This is exactly what the "small scoped windows" philosophy is trying to avoid.

Fix: Add domain field to TaskFile. When loading context for agent X, only inject outputs from tasks whose domain overlaps with X's domain. uiux → motion → frontend is one chain. security → pentest is another. They don't need each other's outputs.
bus.ts:267
Parallel task execution not implemented
routing-matrix.yaml declares parallel-with and tasks have parallel-group, but the CLI runs tasks one at a time. When design-orchestrator and architecture-orchestrator should run simultaneously, they must be manually triggered on two different terminals.

Fix: jodl parallel command — claims all tasks in same parallel-group simultaneously, runs Promise.all() for Claude tasks. For cross-provider parallel, each provider runs their own jodl next concurrently.
bus-commands.ts:new cmd
Embeddings blocked by C++ dependency on Windows
better-sqlite3 + sqlite-vss require native C++ build tools (Visual C++ or MinGW). These aren't installed and are heavy to set up. Phase 2 vector search is deferred. Current keyword search over 5 components is trivially useless.

Fix (no C++ needed): Use Anthropic's claude-3-haiku embedding endpoint or OpenAI text-embedding-3-small via API. Store vectors as JSON file. No native deps. Slower to build but works immediately on any machine.
commands/embed.ts
Pattern corpus too thin — 5 components, all from SARTA
Registry has 5 components: PreloaderScreen, CartDrawer, ProductCard, DragGallery, charRise. All tagged staging quality. No patterns from reference sites. jodl search and jodl compose return near-random results. The agents' "query registry" step is meaningless until corpus reaches 30+.

Fix: Run harvest cycle on 5 reference sites (linear.app, stripe.com, awwwards top 10). Use jodl curate + browser to extract patterns. Target 30 components before next sprint.
jodl-system/sources/
Missing CLI commands documented in README but not implemented
README documents: jodl daemon, jodl cancel, jodl next --force-reclaim. None are in cli.ts or bus-commands.ts. Stale docs mislead new users and other AIs reading the README.
cli.ts / README.md
Import rewrite in jodl vendor is manual step
jodl vendor sarta D:\sarta-for-client copies packages but prints "Manual step required: rewrite imports". The rewrite is fully mechanical (search-replace @jodl/tokens → ./vendor/jodl-tokens/index.js) and can be automated with a regex pass.
commands/vendor.ts:~77
creat-studio and jodlxverse are empty stubs
Both apps exist in the monorepo tree but have no src/, no package.json content beyond placeholders, no Vite config. They're architectural promises not yet delivered.
apps/creat-studio/ apps/jodlxverse/
jodl curate has no auto-scrape — only creates _meta.yaml template
curate.ts has a // TODO Phase 2: Auto-scrape with playwright comment. Currently it only creates a folder + template YAML. The actual inspection of the site is manual. With Playwright MCP available, this could be automated.
commands/curate.ts:~50

Better Architecture — What to Change

🎯 Domain-scoped context injection
The biggest token waste. Every agent getting all prior outputs bloats context, costs tokens, and dilutes focus.
loadSessionContext() → ALL done task outputs → injected into every agent
loadDomainContext(agentRole) → only outputs from same domain chain → injected
// Add to TaskFile schema:
domain: "design" | "architecture" | "implementation" | "security"

// loadSessionContext becomes:
function loadDomainContext(sessionId, domain) {
  return tasks
    .filter(t => t.domain === domain || t.domain === "shared")
    .map(t => loadOutput(t.id))
    .join("\n---\n");
}
🤖 Multi-provider auto-execution
Right now only Claude can auto-run tasks. Gemini and GPT require manual copy-paste. For true autonomous co-work, all 3 need to be wired.
jodl next → only works for claude-code provider
jodl next → auto-routes to Anthropic / Google / OpenAI SDK
switch (task.assignedProvider) {
  case "claude-code":
    return runClaude(task, systemPrompt);
  case "antigravity":
    return runGemini(task, systemPrompt);  // NEW
  case "codex":
    return runOpenAI(task, systemPrompt); // NEW
}
⚡ API-based embeddings (unblock Phase 2)
sqlite-vss needs C++ build tools. Skip that. Use OpenAI text-embedding-3-small or Anthropic's embedding endpoint. Store as flat JSON — no native deps.
sqlite-vss → C++ required → blocked on Windows
openai.embeddings.create() → Float32Array → vectors.json
// embed.ts — replace sqlite-vss with:
const emb = await openai.embeddings.create({
  model: "text-embedding-3-small",
  input: pattern.description,
});
// save to embeddings/vectors.json
// cosine similarity in pure JS — no deps
🔀 True parallel execution
Design and architecture phases should run at the same time. Currently sequential. For Claude-to-Claude parallel, Promise.all() is trivial. For cross-provider, both providers just run jodl next simultaneously.
design runs → finishes → architecture runs (sequential)
design ⇔ architecture simultaneously → implementation waits for both
// jodl parallel command:
const parallelTasks = findParallelGroup(provider, groupId);
await Promise.all(parallelTasks.map(t => runTask(t)));
// dep-check gate: implementation only starts
// when BOTH design + arch are done
🧠 Knowledge packs per agent
Only uiux-master has a knowledge-pack.md. Other agents have no additional knowledge beyond their system prompt. Each agent should have a curated pack: motion-master gets all known motion patterns, frontend-master gets the Codex anti-patterns checklist.
uiux-master: has knowledge-pack.md ✓
others: system-prompt only
Every agent gets: system-prompt + domain knowledge pack + relevant registry slice
📊 Web dashboard for CEO
The terminal watch command is informative but hard to share or glance at. A localhost HTTP server serving an auto-updating HTML page (SSE or polling) would be the CEO's mission control.
jodl watch → terminal-only, monospace grid
jodl watch --web → localhost:4000 → visual board with session/task cards
