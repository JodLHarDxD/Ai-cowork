01

What is this system?

JODL is a multi-AI co-work orchestration system. You submit a brief (e.g. "build a luxury fashion store with RAG-powered search"). The system automatically routes sub-tasks to the best AI provider for each domain, runs them in dependency order (or in parallel), and collects outputs.

Core insight: Different AI providers excel at different domains. Gemini has 2M context and multimodal input — ideal for design research. Claude has the best reasoning for architecture decisions. GPT / Codex runs directly in the codebase. By routing tasks to the right provider, you get better outcomes AND more efficient context usage than using one model for everything.

🤖 Three Providers

Claude (claude-code), Gemini (antigravity), GPT (codex) — each runs on its own platform but shares the same task queue via the filesystem.

📁 Zero Infrastructure

The queue is literally files on disk. Atomic rename() is the distributed lock. Works across Windows, Mac, Linux, WSL — no DB, no server.

🧠 Intelligence Layer

A shared component registry, graph of pairings, feedback loop, and pattern memory accelerates every agent. Built knowledge survives sessions.

✅ What's proven (working now)

  • Full command bus: brief → session → task lifecycle
  • Atomic claim (race-free across concurrent providers)
  • Spawn-chain: orchestrators create child tasks from JSON
  • Claude auto-execution via jodl next

03

Command Bus — The Core Mechanism

The command bus is the backbone of the system. It's a lockless, filesystem-based distributed task queue. No Redis, no Postgres, no broker — just files and an atomic OS rename.

3.1 — The Atomic Lock Pattern

Key insight: On any OS (Windows NTFS, Linux ext4, macOS APFS), rename(A → B) is atomic within the same volume. If two processes try to rename the same file, exactly one wins. This is the distributed lock — no DB transaction needed.

FILENAME pending-a1b2c3d4.yaml
jodl next (atomic rename) →
FILENAME claimed-claude-code-a1b2c3d4.yaml
API call + output →
FILENAME done-a1b2c3d4.yaml
+ a1b2c3d4.out.md
Race condition handled: If two providers both attempt rename(pending-X → claimed-provider-X) simultaneously, exactly one succeeds (OS guarantee). The loser gets an ENOENT error and retries with the next task.

3.2 — Task YAML Anatomy

# D:\.agents\command-bus\active\sess-87af32f2\tasks\pending-7f5fec23.yaml id: 7f5fec23 sessionId: sess-87af32f2 role: frontend-master assignedProvider: codex # who must run this assignedModel: gpt-5 phase: build dependsOn: [a9f2c1ca] # must finish first parallelGroup: 1 # runs parallel with group 1 brief: | Implement the Zara-style Header, Shop, ProductCard, ProductPage... [full task description — can be hundreds of lines]

3.3 — Spawn Chain (Orchestrator → Children)

Every orchestrator's output must end with a JSON block. The bus parses this block and automatically creates child task YAML files. This is how work cascades through the hierarchy.

# Orchestrator output ends with this block (```json): { "phase": "build", "spawn-tasks": [ { "role": "frontend-master", "brief": "Implement the Store page with Zara-style layout...", "depends-on": [], "parallel-group": 1 }, { "role": "backend-master", "brief": "Build FastAPI endpoints for products, cart, auth...", "depends-on": [], "parallel-group": 1 # same group = runs in parallel } ] }
● master-orchestrator (task a9f2c1ca) — outputs JSON spawn block

3.4 — Dependency Resolution

findNextTask(provider) in bus.ts only returns a task if: (1) status is pending, (2) assignedProvider matches current env, (3) ALL ids in dependsOn are in the done set. This means parallel groups execute automatically as their dependencies complete.

# How deps work: design-orchestrator — dependsOn: [] runs immediately architecture-orchestrator — dependsOn: [] runs immediately (parallel) implementation-orchestrator — dependsOn: [design, arch] waits for both security-orchestrator — dependsOn: [implementation] waits + phase=ship gate

Context Injection & Domain Scoping

Every agent gets a system prompt + the outputs of all previously completed tasks injected as context. Before Sprint 1, ALL outputs were injected into ALL agents. This wasted tokens on unrelated work. The new domain scoping keeps context tight.

How it works: domainForRole(role) derives the domain from the role name. loadSessionContext(sessionId, domain) filters done outputs by domain. No new YAML fields required — domain is inferred.

What each agent sees:

Agent Type Domain Sees Doesn't see
master-orchestrator meta ALL outputs from all domains —
design-orchestrator meta ALL outputs —
uiux-master design design domain outputs + all orchestrators implementation, security, ship outputs
frontend-master implementation implementation outputs + all orchestrators design leaf outputs, security leaf outputs
threat-modeler security security outputs + all orchestrators design, implementation leaf outputs

Why this matters: A 10-task session with full outputs might be 40,000 tokens. With domain scoping, a leaf agent in the design domain might only see 8,000 tokens of relevant context. This keeps costs down and reduces hallucinations from unrelated noise.

Context injection path:

# What ends up in the agent's system prompt: [system-prompt.md content] ← role-specific behavior --- ## Prior session outputs ## master-orchestrator (task a9f2c1ca) Brief: Design a Zara-style luxury store... [orchestrator output — always included] ## uiux-master (task b2c3d4e5) Brief: Design the product card component... [same-domain output — included for design agents] # ↑ implementation agents do NOT see this ↑

Brain Memory System

D:\.agents is a git-versioned memory store shared across ALL AI models. Every mistake, proven pattern, and experiment is recorded here. This is what survives between sessions and prevents repeated mistakes.

✅ Confidence Tiers

  • experiments/ — bold untested. Write BEFORE making the move.
  • staging/ — learned once, not proven. Write during session.
  • patterns/ — proven 2+ sessions or projects. Graduate from staging.
  • mistakes/ — confirmed failures. Never repeat. 1 occurrence is enough.

🔑 Integrity Rules

  • ALWAYS git commit D:\.agents after any write.
  • Never delete entries — set confidence: deprecated.
  • Wrong entry: git log D:\.agents → git revert <sha>.
  • Every entry requires frontmatter: confidence, date, project, sessions-validated.

How to Run — Operational Runbook

Prerequisite: At least one API key set. Claude requires ANTHROPIC_API_KEY. For full multi-provider runs, also set GOOGLE_API_KEY and OPENAI_API_KEY.

Step 1 — Set your provider identity

# In PowerShell (Windows): $env:JODL_PROVIDER = "claude-code" # or: antigravity | codex $env:ANTHROPIC_API_KEY = "sk-ant-..." $env:GOOGLE_API_KEY = "AIza..." # for Gemini tasks $env:OPENAI_API_KEY = "sk-..." # for GPT tasks # Verify: pnpm jodl whoami

Step 2 — Submit a brief

# Any provider can submit a brief: pnpm jodl brief "Build a dashboard for analytics with charts, filter, export" # Skip research phase for faster prototypes: pnpm jodl brief --skip-research "Quick prototype: login form" # Output: session ID + master-orchestrator task created # 📋 Session sess-a1b2c3d4 created # ✓ Master orchestrator task spawned

Step 3 — Run your next task (auto)

# The bus finds the next pending task for YOUR provider: pnpm jodl next # See what's waiting without claiming: pnpm jodl next --dry-run # Limit to one session: pnpm jodl next --session-id sess-a1b2c3d4 # ✓ Claimed. 🤖 Running claude-code agent (claude-opus-4-7)... # [streams output to terminal] # ✓ Task done. 🌱 Spawned 5 child task(s).

Step 4 — Check queue status

pnpm jodl status # Shows all sessions, tasks by status (pending/claimed/done), which provider owns each pnpm jodl watch # Live terminal dashboard (run in a dedicated terminal window)
